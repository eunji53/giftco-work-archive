# 기프트코 상품 추천 RAG 학습 기록

## 1. 문서 목적

이 문서는 기프트코 상품 추천 시스템을 구현하면서 진행한 `v0`, `v1`, `v2` 노트북의 학습 목적과 구조를 정리하고, 각 단계에서 확인된 문제점과 다음 보완 방향을 기록하기 위한 문서이다.

각 버전은 운영용 최종 시스템이 아니라 다음 개념을 순차적으로 학습하고 검증하기 위한 실험이다.

```text
v0: 기본 검색과 추천 흐름 구현
v1: 임베딩 기반 검색과 거래데이터 결합
v2: 거래데이터를 Profile 단위로 묶어 검색
```

따라서 특정 방식 자체를 잘못된 것으로 평가하기보다, 각 단계에서 무엇을 구현했고 다음 단계에서 어떤 문제를 개선해야 하는지를 중심으로 정리한다.

---

# 2. 전체 발전 과정

## v0

```text
사용자 질문
→ 상품 검색
→ 유사 거래 검색
→ 거래에서 상품분류 힌트 추출
→ 상품 점수 보정
→ LLM 답변 생성
```

핵심 학습 내용:

- 상품데이터와 거래데이터 전처리
- 검색용 텍스트 생성
- 사용자 질문에서 가격·수량 조건 추출
- 상품 검색 결과에 거래데이터 신호 추가
- 검색 결과를 LLM에 전달하여 답변 생성

---

## v1

```text
사용자 질문
→ LLM·규칙 기반 조건 추출
→ 거래 Row 임베딩 검색
→ 유사 거래 신호 추출
→ 상품 임베딩 검색
→ 가격·수량·카테고리 점수 결합
→ LLM 답변 생성
```

핵심 학습 내용:

- 문장 임베딩 기반 의미 검색
- 과거 거래 Row 단위 임베딩
- 행사월과 실제 주문월의 차이 반영
- 여러 평가 항목을 결합한 재정렬
- 검색과 답변 생성을 분리한 RAG 구조

---

## v2

```text
거래 Row
→ 유사 거래 Profile 생성
→ Profile 임베딩
→ 사용자 질문과 유사한 Profile 검색
→ Profile의 상품·카테고리 신호 추출
→ 현재 상품 재정렬
→ LLM 답변 생성
```

핵심 학습 내용:

- 검색 단위를 Row에서 Profile로 변경
- 반복되는 거래 패턴 집계
- 거래 건수를 검색 신뢰도에 반영
- 대규모 거래데이터 압축 가능성 검토
- Row 검색과 Profile 검색의 차이 비교

---

# 3. v0 검토

## 3.1 v0의 학습 목적

v0의 목적은 상품 추천 RAG의 가장 기본적인 전체 흐름을 구현하는 것이다.

```text
질문을 입력받음
→ 관련 상품을 검색함
→ 과거 거래를 참고함
→ 상품 점수를 계산함
→ 고객용 답변을 생성함
```

복잡한 모델이나 데이터 구조보다 다음 항목을 한 번에 연결해보는 데 의미가 있다.

- 상품 검색
- 거래 검색
- 조건 추출
- 추천 점수 계산
- LLM 응답 생성

---

## 3.2 문제 1: 관련성이 낮은 거래도 정해진 개수만큼 포함됨

현재 방식은 유사도가 충분히 높은 거래가 적어도 `top_k` 개수만큼 결과를 가져올 수 있다.

```text
실제로 관련 있는 거래: 3건
설정된 top_k: 10건
결과: 관련성이 낮은 거래 7건도 함께 사용
```

이 거래에서 추출된 상품분류가 현재 상품 점수에 반영되면 관련 없는 상품군이 추천될 수 있다.

### 보완 방향

상위 개수와 최소 관련도 조건을 함께 사용한다.

```text
최대 10건
AND 최소 유사도 이상
```

또는 최고 점수와의 상대 기준을 적용한다.

```text
1위 점수의 70% 이상인 거래만 사용
```

학습 단계에서는 여러 임계값을 변경하며 결과가 어떻게 달라지는지 비교한다.

---

## 3.3 문제 2: 거래 카테고리 힌트가 너무 넓게 적용됨

유사 거래에서 추출한 카테고리가 `생활용품`, `문구용품`처럼 넓으면 많은 상품이 동일한 보너스를 받을 수 있다.

```text
생활용품 힌트
→ 타월
→ 고무장갑
→ 보석함
→ 칫솔세트
→ 손수건
```

이 경우 카테고리 점수가 상품 간 순위를 구분하지 못한다.

### 보완 방향

카테고리 단계별로 점수를 다르게 준다.

```text
소분류 정확히 일치: 높은 점수
중분류 일치: 중간 점수
대분류 일치: 낮은 점수
```

단순 포함 여부가 아니라 일치 수준을 구분한다.

---

## 3.4 문제 3: 거래와 상품 간 관계가 사라짐

유사 거래를 찾은 뒤 상품분류만 모으면 다음과 같은 원래 관계가 사라진다.

```text
병원 → 위생용품
대학교 → 문구류
기업 행사 → 타월
```

최종적으로는 다음처럼 카테고리 목록만 남는다.

```text
위생용품, 문구류, 타월
```

따라서 어떤 상황에서 어떤 상품군이 선택됐는지 알기 어렵다.

### 보완 방향

거래별 관계를 유지한 상태로 상품 점수를 계산한다.

```text
병원 거래와 유사한 정도
× 병원 거래의 상품분류
```

모든 거래 힌트를 하나로 합치기보다 거래별 신호를 개별적으로 누적한다.

---

## 3.5 문제 4: 가격과 MOQ가 점수로만 처리됨

가격과 최소구매수량이 추천 점수 일부로만 반영되면, 다른 점수가 높은 상품이 조건을 초과해도 상위 결과에 남을 수 있다.

### 보완 방향

조건을 두 종류로 구분한다.

```text
필수 조건
- 판매 상태
- 예산
- 최소구매수량
- 납기
- 재고

선호 조건
- 행사 적합도
- 대상 적합도
- 계절성
- 디자인·스타일
```

필수 조건은 먼저 필터링하고, 선호 조건만 추천 점수로 사용한다.

학습 초기에는 필터와 점수 방식의 결과 차이를 비교해볼 수 있다.

---

## 3.6 문제 5: LLM이 추천 결과를 과도하게 해석할 수 있음

추천 후보가 고객 요구와 정확히 맞지 않아도 LLM이 그 상품에 그럴듯한 추천 이유를 붙일 수 있다.

```text
검색 결과가 부정확함
→ LLM이 자연스러운 문장으로 설명
→ 결과가 실제보다 타당해 보임
```

### 보완 방향

LLM의 역할을 제한한다.

```text
검색·선정: 코드
추천 이유 작성: LLM
```

LLM에는 다음 규칙을 명확하게 전달한다.

- 전달된 상품만 사용
- 상품데이터에 없는 특징을 만들지 않음
- 가격·MOQ·납기를 추측하지 않음
- 적합성이 낮으면 확인 필요라고 표시

# 4. v1 검토

## 4.1 v1의 학습 목적

v1은 v0의 기본 구조에서 임베딩 검색과 구조화된 조건 추출을 추가한 단계이다.

주요 학습 목표는 다음과 같다.

- 상품과 거래를 문장 임베딩으로 변환
- 자연어 질문을 구조화된 조건으로 변환
- 거래 시기와 행사 시기의 차이 반영
- 여러 추천 점수를 결합
- 검색 결과와 생성 결과의 역할 분리

---

## 4.2 문제 1: 거래 Row가 반복되면 검색 결과도 반복됨

v1에서는 거래 한 행을 하나의 임베딩 검색 단위로 사용한다.

```text
거래 1행 = 임베딩 1개
```

동일하거나 매우 유사한 거래가 반복되면 상위 검색 결과가 비슷한 거래로 채워질 수 있다.

```text
동일 구매처
동일 행사
동일 상품
동일 시기
```

이 경우 다양한 구매 패턴보다 반복 거래가 추천에 강하게 반영된다.

### 보완 방향

두 가지 실험을 비교한다.

```text
A. Row 단위 검색 유지
B. 중복 제거 또는 Profile 단위 검색
```

v2는 이 문제를 확인하기 위해 Profile 방식을 실험한 단계로 볼 수 있다.

---

## 4.3 문제 2: 날짜 점수가 의미 유사도를 밀어낼 수 있음

행사월과 주문월의 차이를 반영하는 것은 적절하지만, 날짜 점수 비중이 높으면 같은 월이라는 이유로 관련성이 낮은 거래가 상위에 올라올 수 있다.

예:

```text
질문: 여름 야외 행사 판촉물
검색 거래: 6월에 주문된 일반 문구류
```

월은 비슷하지만 실제 사용 목적은 다를 수 있다.

### 보완 방향

날짜 점수를 보조 신호로 사용한다.

```text
먼저 의미 관련도가 일정 기준 이상인 거래만 선택
→ 그 안에서 날짜 점수로 재정렬
```

권장 순서:

```text
1. 의미 관련도 필터
2. 행사·대상 적합도
3. 날짜·계절 보정
```

---

## 4.4 문제 3: 계절 질문의 검색 기간이 지나치게 넓어질 수 있음

여름을 6~8월로 해석하고, 각 월마다 주문 리드타임을 적용하면 검색 범위가 봄부터 여름까지 넓어질 수 있다.

```text
여름 행사
→ 6월, 7월, 8월
→ 각 행사월 1~3개월 전 주문 반영
→ 3월~8월 거래 영향
```

### 보완 방향

행사 유형이나 상품 제작기간에 따라 리드타임을 다르게 적용한다.

```text
기성품: 짧은 리드타임
인쇄 상품: 중간 리드타임
주문제작 상품: 긴 리드타임
```

초기 학습 단계에서는 고정 리드타임을 사용하되, 해당 가정이 코드에 명시되도록 한다.

---

## 4.5 문제 4: 여러 거래의 정보가 하나의 신호 텍스트로 섞임

유사 거래에서 추출한 구매처, 행사, 대상, 상품분류, 상품명을 하나의 텍스트로 합치면 거래별 연결 관계가 사라진다.

원래 데이터:

```text
병원 + 개원 → 타월
대학교 + OT → 보조배터리
기업 + 박람회 → 볼펜
```

합쳐진 신호:

```text
병원 대학교 기업 개원 OT 박람회 타월 보조배터리 볼펜
```

### 보완 방향

각 거래의 신호를 개별적으로 유지한다.

```python
product_trade_score = sum(
    trade_similarity
    * trade_product_match_score
)
```

즉, 하나의 긴 텍스트로 재임베딩하기보다 거래별 점수를 상품에 누적한다.

---

## 4.6 문제 5: 과거 거래 신호가 고객 질문보다 강하게 작동할 수 있음

현재 상품 최종 점수에서 거래 관련 항목의 합이 고객 질문과 상품 간 직접 유사도보다 높으면, 고객 질문보다 과거 판매 패턴이 추천을 더 강하게 이끌 수 있다.

### 보완 방향

추천 역할을 명확히 구분한다.

```text
현재 상품 검색
→ 추천 후보 생성의 중심

과거 거래 검색
→ 후보 순위 보정
```

예시 가중치 실험:

```text
질문-상품 적합도: 40%
키워드·카테고리 적합도: 20%
거래 패턴: 20%
행사·대상·계절: 15%
기타 품질 신호: 5%
```

정답은 고정되어 있지 않으며 평가 결과를 통해 조정해야 한다.

---

## 4.7 문제 6: 카테고리 부분 문자열 일치가 여러 상품에 동일 점수를 줌

카테고리가 상품 텍스트에 포함되는지만 확인하면 넓은 카테고리에 속한 상품들이 모두 같은 점수를 받을 수 있다.

### 보완 방향

정형 카테고리 컬럼을 직접 비교한다.

```text
대분류 일치 여부
중분류 일치 여부
소분류 일치 여부
```

문자열 검색이 아니라 컬럼 단위로 비교하고, 단계별로 다른 점수를 준다.

---

## 4.8 문제 7: 조건이 없을 때 기본 점수가 최종점수를 높임

고객이 예산이나 수량을 말하지 않았을 때 모든 상품에 동일한 중간 점수를 주면 순위에는 영향이 없지만 최종점수가 높아 보일 수 있다.

```text
최종점수 0.57
≠ 추천 정확도 57%
```

### 보완 방향

조건이 없으면 해당 점수 항목을 계산에서 제외하고 나머지 가중치를 재정규화한다.

예:

```python
active_weights = {
    key: weight
    for key, weight in weights.items()
    if condition_is_available(key)
}
```

이렇게 하면 최종점수의 의미를 더 일관되게 유지할 수 있다.

## 4.9 문제 8: 누락값이 검색 텍스트에 포함될 수 있음

조건 추출 결과가 `None`인 경우 문자열 변환 과정에서 `"None"`이 검색 텍스트에 들어갈 수 있다.

### 보완 방향

텍스트 결합 전에 결측값을 제거한다.

```python
def safe_join(values):
    return " ".join(
        str(value).strip()
        for value in values
        if value is not None
        and not pd.isna(value)
        and str(value).strip()
    )
```

---

## 4.10 문제 9: 날짜 형식과 최신성 처리 부족

거래 날짜가 일반 날짜 문자열이 아니라 엑셀 일련번호일 수 있다.

또한 월만 사용하면 오래된 거래와 최근 거래가 같은 계절 신호로 반영된다.

### 보완 방향

혼합 날짜 형식을 처리한다.

```python
def parse_mixed_date(value):
    if isinstance(value, (int, float)):
        return (
            pd.Timestamp("1899-12-30")
            + pd.to_timedelta(value, unit="D")
        )

    return pd.to_datetime(value, errors="coerce")
```

최신성 점수를 추가한다.

```text
최근 거래: 높은 가중치
오래된 거래: 낮은 가중치
```

---

## 4.11 문제 10: LLM이 과거 거래 상품과 현재 추천 상품을 혼동할 수 있음

LLM에 다음 정보를 동시에 전달하면 혼동 가능성이 있다.

```text
현재 판매 상품 후보
과거 유사 거래에 등장한 상품
```

LLM이 과거 거래 상품을 현재 추천 상품처럼 답변에 포함할 수 있다.

### 보완 방향

현재 추천 가능한 상품을 명확한 ID 목록으로 제한한다.

```json
{
  "allowed_products": [
    {
      "product_id": "P001",
      "product_name": "A 상품"
    },
    {
      "product_id": "P002",
      "product_name": "B 상품"
    }
  ]
}
```

과거 거래 상품은 상품명 대신 집계 정보만 전달한다.

```text
유사 거래에서는 타월류와 위생용품류의 선택 비중이 높았음
```

생성 후 답변에 허용되지 않은 상품이 포함됐는지 검증한다.

# 5. v2 검토

## 5.1 v2의 학습 목적

v2는 거래 Row 단위 검색에서 발생하는 반복과 중복 문제를 줄이기 위해 거래를 Profile 단위로 묶은 단계이다.

주요 학습 목표는 다음과 같다.

- 임베딩 단위를 변경하는 방법
- 거래 집계와 검색의 연결
- 반복 거래를 하나의 패턴으로 표현
- Profile 거래량을 신뢰도로 활용
- Row 방식과 Profile 방식 비교

---

## 5.2 문제 1: Profile 그룹 기준이 너무 세밀함

현재 Profile을 만들 때 구매처 세분류, 상품분류, 주문월, 행사, 대상, 시즌 등 많은 조건이 동시에 사용된다.

모든 값이 같아야 같은 Profile로 묶이기 때문에 실제 거래가 충분히 묶이지 않을 수 있다.

```text
원본 거래: 56,599건
Profile: 원본과 비슷한 수
```

이 경우 Row 방식과 큰 차이가 없어진다.

### 보완 방향

Profile 그룹 기준을 구매 상황 중심으로 단순화한다.

```text
구매처 대분류 또는 중분류
+ 표준화된 행사
+ 표준화된 대상
+ 시즌 또는 주문분기
```

Profile 생성 후 다음 통계를 반드시 확인한다.

- Profile 감소율
- Profile당 평균 거래 수
- Profile당 중앙 거래 수
- 1건 Profile 비율
- 5건 이상 Profile 비율

---

## 5.3 문제 2: 상품분류가 Profile 그룹 기준에 포함됨

상품분류가 Profile을 만드는 조건에 포함되면 다음 거래가 서로 다른 Profile이 된다.

```text
병원 + 개원 + 타월
병원 + 개원 + 텀블러
병원 + 개원 + 칫솔세트
```

하지만 추천 시스템에서 보고 싶은 것은 다음과 같다.

```text
병원 개원 상황에서 어떤 상품군이 많이 선택됐는가?
```

### 보완 방향

상품분류를 그룹 기준에서 제거하고 Profile 내부 결과로 집계한다.

```text
Profile: 병원 + 개원 + 방문객

상품분류 분포
- 타월 35%
- 위생용품 25%
- 텀블러 20%
```

---

## 5.4 문제 3: 대표상품이 빈도순이 아님

대표상품이 데이터에 처음 등장한 순서로 선택되면 실제로 많이 거래된 상품과 대표상품이 다를 수 있다.

### 보완 방향

대표상품을 거래 빈도순으로 생성한다.

```python
representative_products = (
    group["상품명"]
    .value_counts()
    .head(10)
)
```

대표 인쇄방법과 대표 구매처도 최빈값 또는 비율을 함께 저장한다.

---

## 5.5 문제 4: 상품명을 구분자로 합쳤다가 다시 분리함

대표상품을 `/`로 연결하면 상품명 자체에 `/`가 포함된 경우 잘못 분리된다.

### 보완 방향

문자열 하나로 합치지 않고 리스트나 관계 테이블로 관리한다.

```json
[
  {
    "product_name": "컵/텀블러 세트",
    "trade_count": 18
  }
]
```

---

## 5.6 문제 5: 여러 Profile을 하나의 텍스트로 합치면서 관계가 사라짐

Profile 검색 후 여러 Profile의 구매처, 행사, 대상, 상품을 하나의 신호 텍스트로 합치면 Profile별 관계가 다시 사라진다.

```text
병원 + 개원 → 타월
대학교 + OT → 보조배터리
```

가 다음처럼 합쳐질 수 있다.

```text
병원 대학교 개원 OT 타월 보조배터리
```

### 보완 방향

Profile별 상품 신호를 유지하고 현재 상품 점수에 개별적으로 누적한다.

```python
profile_product_score = (
    profile_similarity
    * category_ratio
    * profile_reliability
)
```

---

## 5.7 문제 6: 거래량이 여러 단계에서 중복 반영됨

거래량이 Profile 검색 점수, 신호 추출 점수, 최종 상품 점수에 반복해서 반영되면 과거 인기 상품이 지나치게 강해질 수 있다.

### 보완 방향

거래량의 역할을 한 번만 명확하게 정의한다.

예:

```text
거래량
→ Profile 신뢰도 계산에만 사용
```

로그 변환과 상한을 적용해 대량 거래의 영향이 지나치게 커지지 않도록 한다.

---

## 5.8 문제 7: 거래량 1건 Profile이 많으면 거래량 점수가 의미 없음

Profile이 거의 묶이지 않아 대부분 거래 수가 1건이면 모든 Profile이 비슷한 거래량 점수를 받는다.

### 보완 방향

거래량 점수를 사용하기 전에 Profile 분포를 진단한다.

```text
1건 Profile 비율이 높음
→ 그룹 기준 재설계

충분한 거래가 묶임
→ 거래량 신뢰도 사용
```

---

## 5.9 문제 8: 관련성이 낮은 Profile도 상위 K개에 포함됨

Profile 검색에서도 최소 관련도 없이 상위 개수만 선택하면 관련 없는 Profile이 상품 신호에 포함될 수 있다.

### 보완 방향

v0·v1과 동일하게 최소 관련도 기준을 적용한다.

```text
최대 30개
AND 최소 의미 유사도 이상
AND 최고 점수의 일정 비율 이상
```

---

## 5.10 문제 9: Profile 중앙가격과 중앙 MOQ를 실제 상품 조건처럼 사용함

Profile에는 여러 상품이 들어 있으므로 중앙가격과 중앙 MOQ는 특정 상품의 실제 조건이 아니다.

### 보완 방향

Profile의 가격과 수량은 분포 정보로만 사용한다.

```text
가격 중앙값
가격 25~75% 범위
수량 중앙값
수량 25~75% 범위
```

최종 가격·MOQ 필터는 현재 판매 상품의 실제 값을 사용한다.

---

## 5.11 문제 10: 수량에 맞는 가격 구간 선택이 부족함

대량·중간·소량 가격 중 가장 낮은 값을 사용하면 고객 수량으로 적용받을 수 없는 가격이 선택될 수 있다.

### 보완 방향

수량 구간과 가격을 연결한다.

```text
1~99개 → 소량가격
100~499개 → 중간가격
500개 이상 → 대량가격
```

구간 기준이 데이터에 없다면 별도 컬럼을 추가하거나 가격을 참고값으로만 사용한다.

---

## 5.12 문제 11: 누락된 MOQ를 충족으로 처리할 수 있음

MOQ가 없는 경우 조건 충족으로 처리하면 데이터가 불완전한 상품이 오히려 유리해질 수 있다.

### 보완 방향

세 가지 상태로 분리한다.

```text
충족
미충족
확인 불가
```

`확인 불가`는 중립 처리하고 최종 답변에 확인 필요를 표시한다.

---

## 5.13 문제 12: 연도를 제외한 계절성만 반영함

월만 사용하면 오래된 거래와 최근 거래가 같은 Profile에 포함될 수 있다.

### 보완 방향

계절성과 최신성을 분리한다.

```text
월·분기
→ 계절성

거래일
→ 최신성 가중치
```

# 6. 세 버전의 공통 보완 과제

## 6.1 검색 후보와 최종 추천을 분리

```text
검색 단계
→ 관련 가능성이 있는 상품을 넓게 수집

재정렬 단계
→ 고객 조건과 거래 근거로 순위 결정
```

처음부터 하나의 점수로 모든 것을 해결하려 하지 않고 검색과 재정렬을 구분한다.

---

## 6.2 정형 조건과 의미 조건을 분리

### 정형 조건

- 가격
- MOQ
- 판매 상태
- 납기
- 재고
- 인쇄 가능 여부

### 의미 조건

- 행사
- 구매 목적
- 대상
- 분위기
- 실용성
- 고급스러움
- 계절성

정형 조건은 필터 또는 정확 비교를 사용하고, 의미 조건은 검색과 재정렬에 사용한다.

---

## 6.3 현재 상품과 과거 거래의 역할 분리

```text
현재 상품데이터
→ 추천 가능한 실제 후보

과거 거래데이터
→ 어떤 후보를 우선할지 판단하는 보조 근거
```

과거 거래에 등장한 상품이 현재 판매 상품으로 자동 간주되지 않도록 한다.

---

## 6.4 LLM의 역할 제한

LLM은 다음 역할에 적합하다.

- 고객 질문 구조화
- 행사·대상·용도 표현 정규화
- 최종 추천 이유 작성
- 추가 확인 질문 생성

LLM이 직접 담당하지 않아야 하는 역할:

- 존재하지 않는 상품 추천
- 가격·MOQ·재고 추측
- 검색 후보 외 상품 추가
- 데이터에 없는 특징 생성

---

## 6.5 평가셋 구축

각 버전의 개선 여부를 판단하려면 동일한 질문과 기대 상품군이 필요하다.

예:

| 질문 | 기대 상품군 |
|---|---|
| 병원 개원 답례품 500개 | 타월, 위생용품, 칫솔세트 |
| 대학생 OT 기념품 | 보조배터리, 텀블러, 문구세트 |
| 여름 야외 행사 판촉물 | 선풍기, 양우산, 보냉용품 |
| 고급 VIP 고객 선물 | 브랜드 상품, 고급 텀블러, 세트상품 |

확인 지표:

- Hit@5
- Hit@10
- 기대 상품군 포함률
- 예산 충족률
- MOQ 충족률
- 판매중 상품 비율
- 추천 다양성
- 담당자 적합 평가
- 추천 이유의 데이터 일치 여부

# 8. 권장 v3 방향

v3에서는 세 버전에서 학습한 내용을 다음 구조로 통합할 수 있다.

```text
1. 사용자 질문 입력
   ↓
2. 질문 구조화
   - 구매처
   - 행사
   - 대상
   - 용도
   - 예산
   - 수량

   - 납기
   - 재고
   ↓
4. 현재 상품 하이브리드 검색
   - 임베딩
   - 키워드
   - 카테고리
   ↓
5. 유사 구매 상황 Profile 검색
   ↓
6. Profile별 거래 패턴으로 후보 재정렬
   ↓
7. 유사 상품 중복 제거 및 다양성 조정
   ↓
8. 상위 상품 선정
   ↓

   ↓
10. 추천 상품 ID와 답변 내용 검증
```

핵심 역할은 다음과 같이 구분한다.

```text
현재 상품 검색 = 추천 후보 생성
거래 Profile = 추천 순위 보정
정형 조건 = 필터
임베딩 = 의미 검색
LLM = 질문 구조화 및 설명 생성
```

# 9. 최종 정리

```text
v0
→ 검색, 거래 힌트, 점수 계산, LLM 생성을 연결

v1
→ 임베딩 검색, 조건 추출, 거래 Row 검색, 재정렬을 학습

v2
→ 임베딩 단위를 Row에서 Profile로 변경하고 거래 패턴 집계를 실험
```

현재 확인된 문제는 각 버전의 실패라기보다 다음 학습 단계에서 검증할 항목이다.

```text
검색 결과를 몇 개까지 사용할 것인가?
필수 조건과 선호 조건을 어떻게 분리할 것인가?
과거 거래가 추천에 얼마나 영향을 주어야 하는가?
거래 Row를 그대로 검색할 것인가, Profile로 묶을 것인가?
여러 검색 결과의 관계를 어떻게 유지할 것인가?
LLM이 검색 결과 밖의 내용을 만들지 않게 어떻게 제한할 것인가?
추천이 개선됐는지 어떤 평가 지표로 확인할 것인가?
```

다음 단계에서는 구조를 더 복잡하게 추가하기 전에 동일한 평가셋으로 `v0`, `v1`, `v2`, 개선 버전을 비교하는 것이 중요하다.
